### F-001 · MISMATCHED · Risk: LOW · Confidence: 97%

**Claim (doc):** `docs/architecture.md:87` — "Dev image boots `bin/rails tailwindcss:watch[always] & bin/rails server`."

**Reality (source):** `Dockerfile.dev:35` — The actual CMD is `bash -lc "bin/rails tailwindcss:build && bin/rails db:prepare && bin/rails s -b 0.0.0.0 -p 3000"`. This performs a one-shot build (`tailwindcss:build`), prepares the database, then starts the server — no watch process. The watch command runs only via the `docker-compose.yml` override (`command:` key on line 21), which supersedes the Dockerfile CMD when using Compose.

**Impact:** A developer building and running the dev image directly with `docker run` (not via Compose) will not get live CSS recompilation on file change, contrary to what the documentation implies.

### F-003 · MISMATCHED · Risk: MEDIUM · Confidence: 97%

**Claim (doc):** `docs/modules/auth.md:22-23` — Registration flow step 4 states: "Default workspace: `usage_limits: { scans: 1000, storage_mb: 1024 }`, `subscription_plan: "free"`."

**Reality (source):** `app/controllers/registrations_controller.rb:24-29` — The `Workspace.create!` call explicitly sets `usage_limits` and `usage_consumption` only. `subscription_plan`, `subscription_status`, and `subscription_billing_cycle` are never explicitly assigned in `RegistrationsController#create`; their values come from database column defaults (`"free"`, `"inactive"`, `"monthly"` respectively). The doc implies these are explicitly passed during registration, which is misleading.

**Impact:** A developer modifying the registration flow who relies on the doc may not realise these fields default at the DB level and could inadvertently break them by removing columns or changing defaults without updating the controller.

### F-004 · MISMATCHED · Risk: MEDIUM · Confidence: 98%

**Claim (doc):** `docs/modules/auth.md:22` and `docs/modules/tenancy.md:48` — Registration creates a workspace with slug generated by the `ensure_slug` callback as `"<name-parameterized>-<6-char-alphanumeric>"`.

**Reality (source):** `app/controllers/registrations_controller.rb:26` — `slug: SecureRandom.alphanumeric(10).downcase` — the registration controller explicitly provides a **10-character** pure-random alphanumeric slug, bypassing the `ensure_slug` callback entirely (the callback only triggers when slug is blank). The format is not `<name>-<6-char>` but a 10-char random string.

**Impact:** The documented slug format does not match what is actually generated for user-registered workspaces. Developers expecting a human-readable, name-derived slug will be surprised. Slug collision analysis based on the docs (6-char space) would also be incorrect (10-char space is larger).

### F-005 · MISMATCHED · Risk: HIGH · Confidence: 95%

**Claim (doc):** `docs/architecture.md:95` — "CSP configured … `default-src 'self'`, nonce-based scripts/styles, Stripe frames whitelisted."

**Reality (source):** `config/initializers/content_security_policy.rb:9-10` — `script_src` is set to `:self, :https` (allows all HTTPS scripts, not nonce-only). `style_src` is set to `:self, :unsafe_inline`. While nonces are configured via `content_security_policy_nonce_directives = %w[script-src style-src]` (line 19), the presence of `:https` in `script_src` and `:unsafe_inline` in `style_src` means the policy is significantly weaker than "nonce-based" implies. Any HTTPS-hosted script is allowed for script-src, and inline styles are allowed without nonces for style-src (`unsafe_inline` coexists with nonces but browsers that support nonces may still permit all inline styles when `unsafe_inline` is present).

**Impact:** Developers and security reviewers relying on the doc may believe the CSP is nonce-only strict, when in reality `script_src :https` admits any external HTTPS script and `style_src :unsafe_inline` weakens inline-style protection. This is a security documentation gap.

### F-006 · MISMATCHED · Risk: LOW · Confidence: 92%

**Claim (doc):** `docs/data-models.md:33` — User association `has_many :acting_admin_audit_events` with note "(FK: `acting_admin_id`)"

**Reality (source):** `app/models/user.rb:9` — The association exists and is correctly named, but the `dependent: :destroy` option is also present in source: `has_many :acting_admin_audit_events, class_name: "AdminAuditEvent", foreign_key: :acting_admin_id, inverse_of: :acting_admin, dependent: :destroy`. The doc omits the `dependent: :destroy` detail, which is a meaningful behaviour: deleting an admin user also destroys audit events where they were the acting admin.

**Impact:** A developer who reads the docs and deletes an admin account without awareness of this cascade will lose audit trail data covering actions performed by that admin.

### F-007 · MISMATCHED · Risk: LOW · Confidence: 90%

**Claim (doc):** `docs/modules/tenancy.md:33` — Current workspace resolution "Falls back to the user's oldest membership (`ORDER BY memberships.created_at ASC LIMIT 1`)."

**Reality (source):** `app/controllers/application_controller.rb:27` — `current_user.workspaces.order("memberships.created_at ASC").first`. The `.first` on an ActiveRecord relation applies `LIMIT 1` implicitly. The doc's characterisation is functionally correct but the explicit `LIMIT 1` is not part of the source — it is implied by `.first`. Also, in `SessionsController#create` line 27: `user.workspaces.order("memberships.created_at ASC").pick(:id)` uses `.pick` rather than `.first.id`. This difference in implementation across the two code paths is not mentioned in the docs.

**Impact:** Low; the semantic result is the same. However, documenting `LIMIT 1` as a literal SQL construct is slightly misleading about how the ORM constructs the query.

### F-009 · MISSING_SECURITY · Risk: HIGH · Confidence: 93%

**Claim (doc):** Not mentioned anywhere in `docs/modules/auth.md` or `docs/architecture.md`.

**Reality (source):** `app/controllers/payments_controller.rb:2` — `before_action :authenticate_user!, only: %i[checkout success cancel]`. The `payments#show` action (the pricing/checkout page) is explicitly **not** protected by `authenticate_user!`. The doc (`docs/api.md:27`) lists `GET /payments` under "Authenticated Routes" with the note "(public but reads auth state)" — this is ambiguous but does acknowledge the public nature. However, `docs/modules/payments.md` and `docs/modules/auth.md` never document which payment routes require authentication and which do not, creating an incomplete security picture. Additionally, `POST /payments/webhook` is missing from all auth-flow documentation; the CSRF skip (`skip_forgery_protection only: :webhook`) is mentioned in `docs/architecture.md:96` but the `before_action` exclusion for `webhook` (it is not in the `only:` list) is not documented — webhook does not require authentication, which is correct but undocumented from a security standpoint.

**Impact:** A developer adding new payment routes or extending the controller may not realise that `webhook` deliberately has no authentication (relying solely on Stripe signature verification) and might inadvertently add authentication that breaks webhook delivery.
